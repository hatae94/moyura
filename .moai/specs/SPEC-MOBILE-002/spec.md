## Background (배경)

`moyura` 모노레포는 웹(`apps/web`, Next.js 16.2.6 App Router)을 UI surface 로 삼고, 모바일 앱(`apps/mobile`, Expo ~56.0.6 / RN 0.85.3 / React 19.2.3)은 그 웹을 풀스크린 WebView 로 호스팅한다.

SPEC-MOBILE-001 + 선행 SPEC-WEBVIEW-SHELL-001 이 만드는 상태 (depends-on, live source-of-truth):
- `apps/mobile/App.tsx` — SPEC-MOBILE-001 시점에는 **모놀리식 단일 파일**(WebView + OAuth 인터셉트 + Android 백 + 오버레이 + `runOAuthBridge` 인라인)이며, **선행 SPEC-WEBVIEW-SHELL-001 이 이를 `components/WebViewShell.tsx` + loading/error 오버레이 + 훅(`hooks/useAppLifecycle.ts`/`hooks/useAuthBridge.ts`)으로 행위 보존 추출한다**. 본 SPEC 은 추출 결과 위에 토큰 로직을 얹는다(WebViewShell 추출 자체는 본 SPEC 범위 밖 — SHELL-001 참조).
- `apps/mobile/lib/auth/oauth.ts` / `oauth-bridge.ts` — Google 시스템 브라우저 OAuth 브리지(순수 함수 `isOAuthAuthorizeUrl`/`rewriteAuthorizeRedirect`/`buildWebCallbackUrl` + `launchSocialOAuth`). **보존 대상.**
- `apps/mobile/lib/web-url.ts` — `resolveWebUrl`/`WEB_URL`(`EXPO_PUBLIC_WEB_URL` 가드, `@MX:ANCHOR`). `apps/mobile/lib/env.ts` — `resolve*` 가드 패턴 + `EXPO_PUBLIC_*` 리터럴-키 접근(Expo 인라인).
- `apps/mobile/package.json` — `react-native-webview` 13.16.1, `expo-web-browser`, `expo-linking`, `expo-auth-session` 보유. **`expo-secure-store` 누락.**
- `apps/mobile/app.json` — `scheme: "moyura"`, bundle `com.hatae.moyura`. `apps/mobile/index.ts` — `main`, `registerRootComponent`.

웹 측 (live source-of-truth):

- `apps/web/lib/auth/actions.ts` — `signInWithOAuthAction`, `CALLBACK_URL = "http://localhost:3000/auth/callback"`, `signOutAction`(L69-73: `signOut()` 후 server `redirect("/login")`).

현재의 한계 (이 SPEC 이 해소하는 것): MOBILE-001 의 세션은 **WebView 쿠키 전용**(OD-4)이라 네이티브 코드가 세션 토큰을 들고 있지 않다. 따라서 (a) 앱 재시작/장기 비활성 후 토큰 갱신이 WebView 쿠키 라이프사이클에 전적으로 묶이고, (b) 미래의 네이티브 제품 화면이 백엔드를 직접 호출할 토큰을 가질 경로가 없다. 본 SPEC 은 **세션 권위는 웹에 남기되**(loose coupling), 네이티브에 **토큰 캐시 + 양방향 동기화 핸드셰이크**를 추가해 이 두 한계의 파운데이션을 만든다.

## Goal (목표)

선행 SPEC-WEBVIEW-SHELL-001 이 추출한 `WebViewShell` 컴포넌트 + 합성 가능한 훅(`useAppLifecycle`/`useAuthBridge`) 위에, `expo-secure-store` 기반 네이티브 토큰 캐시(access+refresh)를 도입하여 **토큰 기반 느슨한 결합 세션 파운데이션**을 구축한다. 콜드스타트 시 네이티브가 저장 토큰을 버전드 postMessage 스키마로 웹 브리지에 주입하면, 웹이 `supabase.auth.setSession()` 으로 검증/갱신한 뒤 최신 토큰을 네이티브로 회신하고 네이티브는 SecureStore 를 갱신한다(스플래시가 redirect 플래시를 가린다). resume(AppState active) 시 토큰 재주입+재검증으로 silent refresh 하고, 로그아웃 시 토큰을 클리어한다. 세션 권위(검증/갱신)는 웹이 유지하고 네이티브는 캐시이며, 웹 브리지는 `window.ReactNativeWebView` 가드로 순수 웹 앱 동작을 보장한다(forward-compat 가드레일 3). 버전드/확장 가능 스키마(가드레일 2)도 본 SPEC 이 보유한다. 단일 화면 아키텍처(`expo-router` 미도입)를 유지한다 — 훅 분리(가드레일 1)·generic WebViewShell(가드레일 4)은 선행 SHELL-001 이 충족하므로 본 SPEC 은 그 위에서 확장만 한다.

## Non-Goals (제외 — What NOT to Build)

> [HARD] 본 SPEC 의 명시적 비목표. 최소 1개 이상.

- **네이티브 제품/비즈니스 화면 없음.** 이 SPEC 은 토큰 기반 느슨한 결합의 **파운데이션만** 만든다. 일정/모임/투표 등 실제 네이티브 화면은 이후 SPEC(범위 밖). 장기 비전(WebView ↔ 네이티브 동등)을 인코딩하되 화면은 짓지 않는다.
- **`expo-router` 도입 없음.** 셸은 단일 화면 유지(`App.tsx` 단일 진입). 라이프사이클/브리지 로직의 훅 분리(향후 expo-router 마이그레이션 기계화)는 선행 SPEC-WEBVIEW-SHELL-001 이 이미 수행했다 — 본 SPEC 은 그 훅을 확장만 한다.
- **WebViewShell 컴포넌트화/오버레이 추출/훅 분리 없음.** 이는 선행 SPEC-WEBVIEW-SHELL-001 의 범위다(원래 본 SPEC 의 M1, split 됨). 본 SPEC 은 추출된 `WebViewShell`/`useAppLifecycle`/`useAuthBridge` 를 **확장**할 뿐 추출 자체를 수행하지 않는다.
- **prod 웹 URL / HTTPS 강제 / 앱스토어 배포 연기.** 로컬/dev 우선, env 기반(`EXPO_PUBLIC_WEB_URL` 하드코딩 금지). prod HTTPS 강제는 보안 요구(R-V2)에 문서화하되 배포 자체는 follow-up.
- **Apple / Kakao provider 제외.** MOBILE-001 / SPEC-AUTH-002 와 일관 — Google 만 브리지. Apple 은 Developer Program 미가입.
- **네이티브에서 Supabase refresh 로직 복제 없음.** 네이티브는 `@supabase/supabase-js` 로 직접 토큰을 갱신하지 않는다. 검증/갱신은 **웹의 `supabase.auth.setSession()` 에 위임**한다(중복 갱신 로직 = 투기적 복잡성, OD-3). 네이티브는 토큰 캐시 + 동기화만 담당.
- **신규 웹 라우트/server action/콜백 코드 없음.** 기존 `/login`·`/me`·`signInWithOAuthAction`·`app/auth/callback/route.ts` 를 그대로 재사용한다. 웹 변경은 `window.ReactNativeWebView` 가드로 감싼 **브리지 emit/handler 한 곳**으로 제한(R-T1/R-T4).
- **MOBILE-001 OAuth 브리지 재구현 없음.** `oauth.ts`/`oauth-bridge.ts`/시스템 브라우저 흐름은 **보존**한다(R-V1). 본 SPEC 은 그 위에 세션 토큰 동기화 레이어만 추가한다.

## EARS Requirements

> 모듈 ≤5. 각 요구사항은 acceptance.md 의 AC 와 1:1 대응. `[DELTA]` 마커: `[NEW]` 신규 / `[MODIFY]` 기존 변경 / `[EXISTING]` 변경 없이 의존 / `[EXTEND]` 선행 SPEC-WEBVIEW-SHELL-001 산출물(`WebViewShell`/`useAppLifecycle`/`useAuthBridge`)을 확장 / `[VERIFY]` 수동 검증.
>
> 참고: WebViewShell 컴포넌트화/훅 분리(R-S1~R-S5)는 선행 SPEC-WEBVIEW-SHELL-001 로 split 되었다. 본 SPEC 은 그 결과를 `[EXTEND]` 한다.

### M1. 네이티브 토큰 캐시 + 진입 라이프사이클 (SecureStore + 스플래시)

- **R-N1 (Ubiquitous)** `[MODIFY] apps/mobile/package.json`: The mobile app SHALL declare `expo-secure-store` (token cache backing) AND `expo-splash-screen` (cold-start splash backing for R-N3/R-N4, OD-8) — each installed via `npx expo install` to match the Expo SDK 56 bundled pin — AND a WebView cookie-manager native module (e.g. `@react-native-cookies/cookies`, for R-R4 (c) logout cookie clear; installed via the package manager + autolinked pod, pin noted at implementation time). No hardcoded version in this spec (pins owned by the installer). **`expo-secure-store` 선언이 SPEC-MOBILE-001 OD-4("토큰 저장소 미도입")를 역전한다 — HISTORY 참조.**
- **R-N2 (Ubiquitous)** `[NEW] apps/mobile/lib/auth/token-store.ts`: The app SHALL provide a token store module exposing `loadTokens()`, `saveTokens({access, refresh})`, and `clearTokens()` backed by `expo-secure-store` (OS-keychain encrypted). The refresh token SHALL be stored ONLY in `expo-secure-store` (never `AsyncStorage`/plaintext). **(security — M-1)** Every `setItemAsync` SHALL pass an explicit `keychainAccessible` accessibility option set to a safe value (e.g. `WHEN_UNLOCKED` / `AFTER_FIRST_UNLOCK`, preferably `WHEN_UNLOCKED_THIS_DEVICE_ONLY` to block backup/cross-device restore) — it SHALL NOT rely on the SDK default and SHALL NOT use `ALWAYS`.
- **R-N3 (Event-Driven)** `[MODIFY] App.tsx + [EXTEND] useAppLifecycle`: WHEN the app cold-starts, THEN the shell SHALL display a splash screen, load the last-held `{access, refresh}` from SecureStore, and load `WEB_URL` root into the WebView.
- **R-N4 (Event-Driven)** `[EXTEND] App.tsx + useAppLifecycle`: WHEN the cold-start handshake result is received — `session:synced` OR `session:none` (these are the only two valid cold-start outcomes; `session:cleared` belongs to the logout group R-R2, NOT cold start) — THEN the shell SHALL hide the splash (so any web-side `/login` redirect flash is masked by the splash).
- **R-N5 (State-Driven)** `[EXTEND] App.tsx + useAppLifecycle`: WHILE no tokens are held at cold start, the shell SHALL still load `WEB_URL` root and rely on the existing web guard (`me/page.tsx` `getSession()` empty → `redirect("/login")`) to route to `/login`; the splash SHALL cover the redirect flash. (Unauthenticated cold start.)
- **R-N6 (Unwanted / If-Then)** `[NEW] App.tsx + useAppLifecycle`: IF the cold-start handshake does NOT return a result (`session:synced` or `session:none`) within a bounded timeout (web unresponsive, bridge handler not yet registered, network loss), THEN the shell SHALL hide the splash and fall back to the existing web-guard routing — it SHALL NOT keep the splash indefinitely. Optionally, the shell MAY fall back to a recoverable error state (MOBILE-001 R-U4 retry overlay) instead of a bare web load. (Closes the indefinite-splash gap — B-2.)

### M2. 토큰 동기화 브리지 (버전드 양방향 스키마)

- **R-T1 (Ubiquitous)** `[NEW] apps/mobile/lib/auth/bridge-protocol.ts` + `[MODIFY] apps/web` 브리지 모듈: Native↔Web communication SHALL use a VERSIONED, EXTENSIBLE message schema of shape `{ version, type, ... }` with at least the types `session:restore` (native→web tokens), `session:synced` (web→native latest tokens), `session:none`, `session:cleared`, `resume:revalidate`. Payload SHALL carry access/refresh tokens ONLY (no user profile / `userId`). The schema SHALL be designed so adding token/provider fields later is additive (no breaking change to existing fields). NOTE: if native ever needs the user identifier, it SHALL decode it from the access token (JWT `sub`) — `userId` is intentionally NOT carried over the bridge to minimize PII crossing the JS bridge (strengthens the security posture — relates to OD-4/OD-5).
- **R-T2 (Event-Driven)** `[EXTEND] App.tsx + useAuthBridge`: WHEN the WebView finishes loading the trusted `WEB_URL` origin at cold start, THEN native SHALL inject the stored `{access, refresh}` to the web bridge via a `session:restore` message (using WebView `injectedJavaScript`/`postMessage`).
- **R-T3 (Event-Driven)** `[MODIFY] apps/web` 브리지 모듈 (uses the browser client `lib/supabase/client.ts` — NEW client-side wiring, currently unused — see Background B-1): WHEN the web bridge receives `session:restore`, THEN the web SHALL call `supabase.auth.setSession({access_token, refresh_token})` (browser context) to validate/refresh. The web SHALL obtain the refreshed `{access_token, refresh_token}` from the `setSession()` return value (`data.session`) — or, if that is unavailable, via an `onAuthStateChange` subscription that fires on the refresh (mechanism choice = OD-9). Outcomes:
  - IF valid or refreshed, THEN the web SHALL post the LATEST `{access, refresh}` back to native as `session:synced` (tokens only, no `userId`) and routing SHALL land on the main surface;
  - IF the refresh token is empty/expired, THEN the web SHALL post `session:none` and the existing web guard SHALL redirect to `/login`;
  - IF `setSession()` throws / fails with a network or runtime error (neither valid nor empty), THEN the web SHALL post `session:none` (the throw/exception outcome is consolidated to `session:none` — no separate type is introduced, so the guaranteed type set in R-T1 stays the 5 enumerated types) so native hides the splash and falls back to the login route — the error path SHALL NOT leave the handshake unresolved (B-2).
  - Authority note (M-5): the authority of validity is the `setSession()` refresh success (and the backend JWKS guard), NOT `me/page.tsx` `getSession()` — the latter is only a cookie-presence routing guard that does not verify the signature. The browser `setSession()` cookie write SHALL be consistent with the server `getSession()` read in `me/page.tsx`.
- **R-T4 (Unwanted)** `[MODIFY] apps/web` 브리지 모듈: The web bridge emit/handler SHALL be guarded by `if (window.ReactNativeWebView)` so that, when loaded as a standalone browser app, the web behaves identically and no native dependency leaks into the pure-web path (forward-compat guardrail 3).
- **R-T5 (Event-Driven)** `[MODIFY] App.tsx + token-store`: WHEN native receives `session:synced` (via `onMessage`), THEN native SHALL update SecureStore with the latest `{access, refresh}`; web SHALL own routing (native SHALL NOT reload the WebView to change route).
- **R-T6 (Unwanted)** `[EXTEND] useAuthBridge`: IF the WebView is NOT on the trusted `WEB_URL` origin, THEN native SHALL NOT inject tokens (origin allowlist — never inject into an arbitrary navigated page). The allowed-origin set SHALL be derived from the `EXPO_PUBLIC_WEB_URL` host and be consistent with the OD-7 host decision (localhost-consistent emulator setup recommended). This origin allowlist applies to BOTH cold-start injection (R-T2) AND resume re-injection (R-R1). Token values SHALL NOT be logged, and injected JS SHALL prefer `postMessage` channels over raw string interpolation of token values.
- **R-T7 (Unwanted / If-Then)** `[NEW] App.tsx + useAuthBridge + apps/web` 브리지 모듈: IF the web bridge handler is NOT yet registered when native injects `session:restore` (the `onLoadEnd`↔handler-registration race noted in plan.md), THEN native SHALL retry injection (bounded retries) OR the web SHALL buffer the message and ack on handler registration — so the `session:restore` message is NOT silently lost. (Closes the injection-race gap — B-2; bounded retries also feed the R-N6 timeout.)
- **R-T8 (Unwanted, security — C-1 / H-1)** `[NEW] apps/mobile/hooks/useAuthBridge.ts + apps/web` 브리지 모듈: The native→web token injection SHALL use a SPECIFIC trusted-origin `targetOrigin` (the `EXPO_PUBLIC_WEB_URL` origin literal) — it SHALL NOT use wildcard `"*"` (which broadcasts tokens to every listener in the document). The web bridge SHALL authenticate every inbound token-bearing message BEFORE acting on it: it SHALL validate the message origin (`event.origin` === trusted WEB_URL origin) AND verify message authenticity via a shared per-session secret (a one-time nonce or message HMAC established native↔web at cold-start over the trusted channel — mechanism = OD-11). It SHALL REJECT any `session:restore` (or other token-bearing message) that is foreign-origin OR unauthenticated (schema-valid-but-unsigned). Symmetrically, native SHALL only accept inbound bridge messages it can authenticate (same nonce/HMAC). Schema-shape validation alone (existing `parseInboundMessage`) is NOT sufficient — it validates form, not sender identity. (Defeats same-page third-party script / supply-chain / XSS forging `session:restore` for session fixation, and token harvest via a rogue `message` listener.)
- **R-T9 (Unwanted, security — C-2)** `[NEW] apps/mobile/components/WebViewShell.tsx + apps/mobile/hooks/useAuthBridge.ts`: The WebView SHALL be locked to the trusted `WEB_URL` origin: `originWhitelist` SHALL be restricted to the trusted origin (NOT the default `["http://*","https://*"]`), AND `onShouldStartLoadWithRequest` SHALL DENY in-WebView loads of any non-trusted top-level origin (delegating non-trusted navigation to the external browser via `Linking.openURL`), WHILE PRESERVING the existing OAuth authorize→system-browser interception. Token injection (cold-start R-T2 AND resume R-R1) SHALL re-validate the LIVE current WebView origin at the moment of injection (NOT a stale cached `currentUrlRef`) so there is no TOCTOU window. (Ensures injected tokens can never reach an attacker-controlled page; combined with R-T8 specific `targetOrigin`, an off-origin page receives nothing even if reached.)

### M3. Resume 재검증 + 로그아웃 클리어

- **R-R1 (Event-Driven)** `[EXTEND] useAppLifecycle`: WHEN `AppState` transitions to `active` (resume) AND tokens are held, THEN native SHALL FIRST pass the R-T6 origin allowlist (if the current WebView origin is NOT the trusted `WEB_URL`, native SHALL NOT inject — the user may have navigated to a third-party page before backgrounding; H-3), AND native SHALL debounce / compare against the previous `AppState` to suppress duplicate `active` transitions (avoid duplicate token injection / refresh contention; B-2). THEN native SHALL inject the stored tokens + a `resume:revalidate` signal via `injectedJavaScript`, prompting the web to silently refresh and post the latest tokens (`session:synced`); native SHALL then update SecureStore. The web SHALL own routing (no native reload).
- **R-R2 (Event-Driven)** `[MODIFY] apps/web` 브리지 모듈: WHEN logout occurs on the web, THEN the web SHALL post `session:cleared` to native (guarded by `window.ReactNativeWebView`). Because `signOutAction` is a Server Action that ends with a server `redirect("/login")` (no client-JS window after redirect — H-2), the emit SHALL occur at a point that does NOT race the redirect: emit ONCE from the client after arriving at `/login` (window.ReactNativeWebView guard on `/login` mount), OR from the `/me` logout button's client handler BEFORE invoking `signOut` (emit point = OD-10). The emit SHALL NOT be placed inside the Server Action body.
- **R-R3 (Event-Driven)** `[MODIFY] App.tsx + token-store`: WHEN native receives `session:cleared`, THEN native SHALL clear the SecureStore tokens (`clearTokens()`).
- **R-R4 (Unwanted, security — H-2 / M-3 / device-verif cookie resurrection)** `[NEW] apps/mobile/lib/auth/bridge-protocol.ts + token-store.ts + apps/mobile/hooks/useAuthBridge.ts (cookie clear) + apps/web` 브리지 모듈: A missed `session:cleared` SHALL NOT leave a stale refresh token in SecureStore, AND a non-persisted web-side cookie deletion SHALL NOT resurrect the session on the next cold start. (a) WHEN native receives `session:none` from the handshake (web found no valid session / refresh expired / setSession failed), THEN native SHALL ALSO clear the SecureStore tokens (`clearTokens()`) — not merely skip saving (closes M-3: the inbound `none` action must clear, not no-op). (b) The logout `session:cleared` emit SHALL be delivered reliably so a single dropped emit does not grant persistent access: via ack/retry, OR rendered idempotent by the `session:none`→clear path in (a) re-clearing on the next cold-start. (c) WHEN native receives `session:cleared` (explicit logout), THEN native SHALL — IN ADDITION TO `clearTokens()` — clear the WebView cookies for the trusted `WEB_URL` origin (via a cookie-manager API that clears BOTH the WKWebView cookie store AND `NSHTTPCookieStorage`), so a deletion that failed to persist web-side cannot resurrect the session on the next cold start. This cookie-clear SHALL be scoped to `session:cleared` ONLY — it SHALL NOT fire on `session:none` (which also occurs on a `setSession` network-throw per R-T3; clearing cookies there could destroy a valid cookie session on a transient error — for that path the web remains the session authority). Rationale (measured, plan.md § 2026-06-11): WKWebView/`NSHTTPCookieStorage` cookie deletions originating from a server `Set-Cookie` are NOT reliably persisted to `binarycookies`; native-side cookie clear is defense-in-depth turning a bounded-resurrection window (≤ `jwt_expiry` = 3600s) into zero resurrection. Implementation dependency: a cookie-manager native module (e.g. `@react-native-cookies/cookies`, installed via the package manager + autolinked pod; pin noted at implementation time — declared in R-N1). (Defeats session resurrection via a stale refresh token re-injected OR a non-persisted cookie deletion on the next cold start.)

### M4. 보존 + 검증 (Preservation / Verify)

- **R-V1 (Ubiquitous)** `[EXISTING] oauth.ts / oauth-bridge.ts / App.tsx`: The existing Google OAuth system-browser bridge AND email/password login inside the WebView SHALL continue to work without regression; this SPEC SHALL NOT reimplement or remove the SPEC-MOBILE-001 OAuth flow.
- **R-V2 (Ubiquitous)** `[NEW] security constraints`: As tokens cross the JS bridge (security-sensitive), the design SHALL enforce: refresh token only in `expo-secure-store` with an explicit safe `keychainAccessible` (R-N2); native-side origin allowlist before injection (R-T6); **web-side inbound message authentication (origin + nonce/HMAC) and specific `targetOrigin` (R-T8); WebView origin locking + live-origin re-validation (R-T9); reliable logout / `session:none`→clear + logout WebView cookie clear (R-R4 (a)(b)(c))**; no token logging; prod requires HTTPS for the web origin (dev `localhost` exception). The expert-security OWASP follow-up review (OD-5) was performed (security-review.md, FAIL) and its CRITICAL/HIGH findings are tracked by R-T8/R-T9/R-R4/R-N2; a re-review SHALL confirm closure at Run phase.
- **R-V3 (Event-Driven, manual)** `[VERIFY]`: WHEN a human runs the shell on an emulator/device, THEN: (a) cold start with a valid stored session SHALL land on the main surface without showing `/login` (splash masks any flash); (b) cold start with no/expired session SHALL land on `/login`; (c) login (Google + email) SHALL post tokens that native persists to SecureStore; (d) resume after token expiry SHALL silently refresh and update SecureStore; (e) logout SHALL clear SecureStore — all without regressing the SPEC-MOBILE-001 WebView/OAuth behavior.
