# Interview: 스토어 정책 대응 — 신고·차단 (SPEC-SAFETY-001)

## 배경

스토어 출시 정책 요건 대응:
- Apple App Review Guideline 1.2 (UGC 앱: 신고 메커니즘 + 가해 사용자 차단 수단 필수)
- Google Play UGC 정책 (신고/차단 요건 동일 취지)
- 회원 탈퇴 요건은 별도 SPEC(SPEC-ACCOUNT-001)에서 다룸

## Round 1: 범위 및 구성 (4문항 단일 라운드)

Question: 세 기능(신고, 차단, 회원 탈퇴)을 SPEC 문서로 어떻게 구성할까요?
Answer: 2개로 분리 — 신고+차단은 하나의 UGC 모더레이션 플로우로 SPEC-SAFETY-001, 회원 탈퇴는 독립 SPEC-ACCOUNT-001

Question: 멤버를 차단하면 어떤 범위까지 컨텐츠를 가릴까요?
Answer: 모든 UGC 숨김 — 채팅 메시지 + 그 멤버가 생성한 기타 가시적 컨텐츠를 내 화면에서 숨김. 차단 진입점은 신고 직후 + 멤버 목록 어디서나. 차단은 신고와 독립적으로 동작.

Question: 회원 탈퇴 시 데이터는 어떻게 처리할까요? (SPEC-ACCOUNT-001 소관)
Answer: 즉시 삭제 + UGC 익명화 — 계정/개인정보(프로필, 이메일, FCM 토큰) 즉시 삭제, 모임 내 기록은 '탈퇴한 사용자'로 익명화 보존

Question: 접수된 신고는 어떻게 처리되도록 할까요?
Answer: 저장 + 신고자 즉시 숨김 — 신고를 사유와 함께 DB 저장, 신고자 화면에서 해당 컨텐츠 즉시 숨김 + 차단 유도. 관리자 검토 화면은 범위 제외(운영 검토는 DB 조회 기반 수동 프로세스, 스토어의 '24시간 내 조치' 요건은 운영 절차로 대응).

## Round 2: 계획 검토 게이트 확정 결정 (2026-07-02)

계획(plan.md) 검토 게이트에서 사용자가 확정한 결정 4건. HISTORY(spec.md) "사용자 확정 결정 4건"의 온-디스크 근거.

1. **지출 표시 방식** — 차단 대상 expense 행을 완전 숨김하지 않고 **목록에 유지 + 작성자 표시만 '차단한 멤버' 마스킹**으로 확정. 근거: 행을 제거하면 뷰어가 보는 표시 합계와 항목 목록의 합이 어긋나 원장 정합성 혼란. 정산 계산(balance/transactions/total)은 전체 원본 포함(REQ-FLT-003). Round-1 "모든 UGC 숨김" 답변의 예외 — 지출만 정산 무결성을 위해 마스킹 방식으로 확정(사용자 승인).
2. **차단 해제 UI 배치** — 전용 설정 라우트 **미신설**, 기존 프로필/설정 화면 내 **"차단한 멤버" 섹션**으로 조회+해제 제공(REQ-BLK-004). 최소 배치.
3. **고아 정리 소유권** — block/report 고아 행 정리 소유권 = **SPEC-ACCOUNT-001**, 구현 순서 **SAFETY 테이블 선행**(§3.5, R-10).
4. **탈퇴 사용자 report 행 처리** — 탈퇴(ACCOUNT-001) 사용자 연관 `report` 행은 ACCOUNT `deleteAccount` 트랜잭션에서 **삭제**(감사 보존 테이블 이관 없음). 탈퇴 후 대상 부재로 운영자 수동 검토 조치 불능이라 보존 가치 없음.
