## 제품 개요

**한 줄 정의**: moyura는 일반 소비자(B2C)를 위한 생산성/유틸리티 도구다.

- **유형/도메인**: 생산성 / 유틸리티 / 도구 (productivity / utility tool)
- **세부 하위 도메인**: **TBD** — 아직 구체화되지 않음.
  - 정제 필요: "어떤 작업/맥락을 더 효율적으로 만드는 도구인가?"를 명시하면 하위 도메인이 결정된다.
  - 예: 일정/메모/태스크 관리, 개인 데이터 정리, 빠른 입력/캡처, 변환/계산 유틸 등 — **현재 미정**.
  - 하위 도메인이 확정되면 별도 SPEC으로 기능을 구체화한다(`/moai plan` 권장).

- **범위 성격**: 처음부터 **비교적 넓은 기능셋**을 목표로 한다 (좁은 단일 기능 MVP가 아님).
- **구체 기능 목록**: **TBD** — 넓은 기능셋을 지향한다는 방향만 확정되었고, 개별 기능은 아직 정의되지 않았다.
- **구체화 방식**: 각 기능은 확정 시점에 개별 SPEC 문서(`.moai/specs/SPEC-*`)로 구체화한다.
  - 인프라 배선([`SPEC-ENV-SETUP-001`](../specs/SPEC-ENV-SETUP-001/spec.md))·인증([`SPEC-AUTH-001`](../specs/SPEC-AUTH-001/spec.md)) 위에 **"모임(moim)" 도메인이 첫 제품 축으로 구체화되어 구현됨**: 모임 CRUD·멤버십, 초대 링크+게스트 참여, 실시간 채팅, 일정/장소/투표(이벤트 트라이어드), 투표 마감·자동 확정, 멤버 강퇴/소유권 양도/정원, 회비/정산, FCM 푸시(in-progress). 상세 구현 현황은 [tech.md](./tech.md) "구현됨 vs 계획됨" 표를 단일 출처로 참조한다.
- **주의(honest scope)**: "넓은 기능셋"은 동시에 많은 미확정 영역을 의미한다. 제품의 상위 하위 도메인(아래 TBD)은 여전히 미확정이며, 현재 구현된 "모임" 기능군이 제품 전체 방향을 확정한 것은 아니다. 기능 우선순위·범위 경계는 도메인 확정 후 재검토 대상이다.

## 플랫폼 전략 — RN 웹뷰 하이브리드

moyura는 **React Native(Expo) 웹뷰 하이브리드** 아키텍처를 채택한다. 세 앱의 역할은 다음과 같다.

| 구성 요소 | 역할 | 비고 |
|-----------|------|------|
| `apps/mobile` (Expo RN) | **앱 셸 / 네이티브 래퍼** — 콘텐츠를 WebView로 호스팅 | 네이티브 권한·배포·디바이스 연동 담당 |
| `apps/web` (Next.js) | **메인 UI 표면(shared content surface)** — 실제 사용자 인터페이스 | 모바일/웹 양쪽이 공유하는 단일 콘텐츠 소스 |
| `apps/backend` (NestJS) | **백엔드 API** — 두 프런트엔드가 공통으로 소비 | REST 기반 |

**관계 흐름**:

```
mobile (네이티브 셸) ── WebView ──▶ web (Next.js, 메인 UI 표면)
                                          │
mobile ─┐                                 │
        ├───────── HTTP REST ────────────▶ backend (NestJS) ──▶ (DB)
web    ─┘
```

- 네이티브 앱은 **얇은 셸**로, 핵심 화면을 직접 네이티브로 만들지 않고 web 콘텐츠를 WebView로 띄운다.
- web은 **단일 UI 표면**이므로, 한 번의 UI 작업이 모바일과 웹 양쪽에 반영된다(중복 구현 최소화).
- 두 프런트엔드 모두 동일한 backend REST API를 소비한다.

### 함의 (Implications)

- **장점**: UI를 web 한 곳에서 관리 → 모바일/웹 일관성 확보, 기능 출시 속도 향상.
- **제약/주의**:
  - WebView 하이브리드 특성상 순수 네이티브 대비 일부 성능·제스처·오프라인 경험에 제약이 있을 수 있다 — 구현 시 검증 대상.
  - "어떤 기능을 네이티브 셸에서 직접 처리하고(권한, 푸시, 딥링크 등) 어떤 기능을 WebView에 위임할지"의 경계는 점진적으로 확정되는 중이다 — 현재는 네이티브가 인증(Google Sign-In)·토큰 세션 브리지·푸시 등록/탭·딥링크(`moyura://`)를 직접 처리하고, 화면 UI 는 web 표면을 WebView 로 위임한다.
  - 현재 코드 상태: 하이브리드는 더 이상 의도가 아니라 **구현된 아키텍처**다. `apps/mobile`은 expo-router 네이티브 네비게이션 골격 + 라우트별 WebView 래퍼 + 토큰 기반 세션 브리지(SecureStore + nonce postMessage)를 갖추고, 핵심 플로우가 iOS 시뮬레이터/디바이스로 검증되었다. 단, 실기기 FCM 종단 수신 등 일부 항목은 device-gated(검증 대기) 상태다. 상세는 [structure.md](./structure.md) "RN 웹뷰 하이브리드"·[tech.md](./tech.md) 참조.

## 비범위 / 가정 (Non-Goals & Assumptions)

**현재까지 명시된 비범위**:
- B2B/엔터프라이즈 기능, 개발자 대상 기능은 범위 밖이다.
- **B2C 인증(로그인)은 구현됨**: [`SPEC-AUTH-001`](../specs/SPEC-AUTH-001/spec.md)(completed)에서 Supabase 기반 email/pw 로그인 + 소셜 OAuth 스캐폴드가 구축되었다(authn-only). 단, 인가(RBAC)·이메일 확인/재설정·실제 소셜 키 등 인증 후속 과제와 구체 제품 기능은 여전히 TBD다.

**가정**:
- 일반 소비자가 별도 온보딩 없이 사용 가능한 수준의 UX가 요구된다(가정 — 검증 필요).
- web을 단일 UI 표면으로 두는 하이브리드 전략이 유지된다는 전제 위에서 이후 기능 SPEC이 작성된다.

**확정 필요(요약 TBD 목록)**:
- 세부 하위 도메인
- 타깃 사용자 페르소나/사용 맥락
- 핵심 가치/해결 문제(value proposition)
- MVP 개별 기능 목록 및 우선순위
- 네이티브 셸 ↔ WebView 책임 경계
